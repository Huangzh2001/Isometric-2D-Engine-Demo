# 项目架构说明（中文）

## 1. 文档目的

本文档用于说明本项目当前的技术架构、模块职责、数据流与开发边界，服务对象包括：

- 后续接手的开发者
- 需要在受控边界内修改代码的 AI
- 需要快速理解项目结构的维护者

本文档描述的是**推荐的架构视图**，并不要求当前代码已经完全达到该结构。后续重构应以此为方向逐步推进，并坚持“小步重构、行为不变、逐步验证”的原则。

---

## 2. 项目定位

本项目是一个基于浏览器与本地轻量服务运行的等距房间编辑与展示系统。当前由两个一级子系统构成：

1. **主程序（Main Program / Game / Scene Program）**
   - 负责场景使用、物品摆放、人物运动、图层排序、光影渲染、场景保存与恢复、素材导入后的实际消费。

2. **素材编辑器（Asset / Prefab Editor）**
   - 负责制作主程序可直接消费的 prefab 素材，包括 voxel、sprite、anchor、offset、scale 等参数配置与导出。

这两个子系统是**两个入口不同、职责不同、通过统一 prefab 数据协议连接的子系统**。

---

## 3. 一级结构

### 3.1 主程序

主程序面向“场景搭建与运行”，主要承担以下职责：

- 将 prefab 实例化到场景中
- 管理物品的空间位置与前后关系
- 处理人物移动、碰撞与遮挡
- 处理光照与阴影的可选渲染
- 保存、读取与恢复场景
- 管理日志与调试输出
- 导入并管理外部素材来源

### 3.2 素材编辑器

素材编辑器面向“素材生产”，主要承担以下职责：

- 编辑 voxel 结构
- 导入 sprite 图片
- 调整 scale / offset / anchor
- 预览素材在等距场景中的显示效果
- 导出统一 prefab 数据

### 3.3 两者关系

素材编辑器与主程序的关系不是 UI 级别的耦合关系，而是**通过 prefab 数据协议连接**。

即：

- 编辑器负责生成标准 prefab
- 主程序负责读取并消费标准 prefab

因此，二者之间最重要的连接点是：

## **Prefab 数据格式（素材协议）**

## 4. 主程序的二级结构

### 4.1 物品放置与排序系统

负责：

- 将 prefab 实例放入场景
- 管理实例的位置、尺寸、旋转、命名与偏移
- 处理物体之间的前后关系
- 支持选中、拖动、删除、修改属性等编辑行为

该系统回答的问题是：

> 场景中有哪些对象、它们位于哪里、它们应当如何参与渲染排序。

这里的“图层”不是简单 z-index，而是等距视角下基于占地、锚点、遮挡与排序规则的统一空间逻辑。

### 4.2 人物运动、碰撞与人物相关图层系统

负责：

- 角色位置更新
- 行走方向与动画状态切换
- 与场景物体的碰撞检测
- 可通行区域判断
- 人物与物体之间的前后遮挡关系
- 人物代理体积、脚底点、人物阴影相关的空间计算

该系统与物品系统紧密协作，但应保持职责分离：

- 物品系统关心场景中“有什么”
- 角色系统关心角色“如何在其中运动并与之发生空间关系”

### 4.3 光影系统

负责：

- 光源定义与管理
- 光照参数（强度、颜色、范围、角度等）
- 物体阴影
- 人物阴影
- 光影可视化开关

光影系统应被视为：

> 主渲染流程上的可选增强模块

而不是所有基础逻辑的前置依赖。该系统必须允许关闭，以便调试与性能控制。

### 4.4 场景存储与恢复系统

负责：

- 场景序列化
- 场景反序列化
- 浏览器本地持久化（如 localStorage）
- 默认场景读取
- 场景文件保存 / 加载
- 导入 / 导出场景 JSON

该系统的本质职责是：

> 将运行态场景状态转化为可持久化、可恢复的数据，并在之后正确恢复回运行态。

此系统保存的是**scene（实例状态）**，不是 prefab（素材定义）。

### 4.5 日志系统

负责：

- 记录用户操作
- 记录状态变化
- 记录 IO / 接口结果
- 输出调试信息与错误信息
- 提供问题定位所需的可观察性

日志系统不参与业务决策，只负责记录与辅助调试。

### 4.6 素材管理与导入系统

负责：

- 管理 prefab 库
- 导入 Habbo 资源
- 导入编辑器导出的 prefab
- 维护素材索引与资源浏览入口
- 将不同来源的资源转换为主程序可用的统一结构

长期目标应是建立统一的素材接入层，而不是把某个单一来源写死为唯一标准。

### 4.7 输入与交互控制层

负责：

- 监听鼠标 / 键盘 / 滚轮 / 拖拽输入
- 识别当前编辑模式
- 将原始输入转换为标准动作
- 将动作分发给状态层、逻辑层与渲染层

### 4.8 渲染与更新调度层

负责：

- update loop
- render loop
- 各子系统调用顺序
- 可选模块的挂接顺序
- 每一帧的更新与绘制节奏

### 4.9 核心数据模型层

负责统一定义：

- prefab
- scene
- instance
- light
- player
- asset library item

这一层负责回答：

> 系统中的关键对象到底长什么样。

## 5. 素材编辑器的二级结构

### 5.1 voxel 编辑模块
负责 voxel 结构编辑。

### 5.2 sprite 编辑模块
负责图片导入、scale、offset、anchor 等配置。

### 5.3 prefab 导出模块
负责将编辑结果导出为统一 prefab 数据。

### 5.4 编辑器状态模块
负责管理当前编辑对象、模式、预览状态。

### 5.5 预览渲染模块
负责在编辑器中实时预览素材的显示效果。

## 7. 核心数据流

### 7.1 编辑器到主程序的数据流

```text
素材编辑器
    ↓
生成 prefab JSON
    ↓
保存到 prefab 库
    ↓
主程序读取 prefab 索引
    ↓
用户选择 prefab
    ↓
在场景中实例化摆放
```

### 7.2 主程序内部的数据流

```text
用户输入
    ↓
交互控制层
    ↓
状态更新 / 逻辑计算
    ↓
渲染排序与图层判断
    ↓
可选光影增强
    ↓
最终画面输出
```

### 7.3 场景保存恢复的数据流

```text
当前运行状态
    ↓
序列化为 scene 数据
    ↓
写入本地文件 / 浏览器存储
    ↓
重新加载
    ↓
恢复场景实例与配置
```

## 8. 开发边界与规则

### 规则 1
主程序不得依赖编辑器内部 UI 逻辑。两者只能通过 prefab 数据协议连接。

### 规则 2
新增素材来源时，应先接入素材管理与导入系统，再转换为统一 prefab，而不是直接向主程序核心逻辑注入特殊处理。

### 规则 3
排序、碰撞、遮挡属于统一空间逻辑，不应在多个模块中各写一套不兼容规则。

### 规则 4
scene 与 prefab 必须严格区分：

- scene = 实例状态
- prefab = 素材定义

### 规则 5
光影系统必须允许关闭，不能成为主程序其他基础逻辑的硬依赖。

### 规则 6
重构时优先“抽离与集中”，避免在同一轮改动中同时改变行为与结构。

### 规则 7
对 AI 的任务应始终限定边界：

- 改哪些文件
- 不改哪些模块
- 数据结构是否允许变动
- 验收标准是什么

## 9. 当前阶段的重构策略

当前项目应采用以下策略推进：

1. 先建立文档与边界
2. 小步抽离独立系统
3. 每轮只处理一个模块
4. 每轮改动后立即进行最小回归验证
5. 行为稳定后再考虑更深层的结构整理

第一轮已建议从**场景存储与恢复系统**开始，而不是先处理高耦合的光影系统。
