# 场景存储与恢复系统重构任务说明

## 1. 目标

将主程序中的“场景存储与恢复系统”抽离为独立模块，集中管理 scene 的保存、读取、默认场景加载与本地持久化相关逻辑，在不改变现有运行行为的前提下提升代码结构清晰度。

本轮只处理：

## **Scene Storage / Restore System**

---

## 2. 本轮允许的改动

### 2.1 新增目录与文件

新增：

```text
src/scene-storage/scene-storage.js
```

### 2.2 需要集中到新模块中的逻辑

应尽量集中以下逻辑：

- scene 序列化
- scene 反序列化
- 默认 scene 创建 / 读取
- scene 保存
- scene 加载
- 本地持久化（如 `localStorage`）
- 调用以下接口的逻辑：
  - `/api/scenes/save`
  - `/api/scenes/load`
  - `/api/scenes/default`
- scene JSON 导入 / 导出辅助逻辑
- 场景重置 / 基于世界参数创建新场景的辅助逻辑

### 2.3 接线要求

- 原有调用位置必须继续可用。
- 如有需要，可以保留兼容包装函数。
- 如有必要，可以在 `index.html` 中新增新文件的 `<script>` 引用。
- 脚本加载顺序只能做最小必要调整。

---

## 3. 本轮禁止的改动

本轮**不得**进行以下改动：

- 不修改光影系统
- 不修改人物移动系统
- 不修改碰撞行为
- 不修改物品排序行为
- 不修改 prefab 数据结构
- 不修改 scene 数据结构
- 不修改 API 路径
- 不修改字段名
- 不修改主程序 UI 行为
- 不修改编辑器逻辑
- 不额外重命名无关模块
- 不做大规模目录迁移
- 不将项目改造成 ES module / import-export 架构
- 不顺手进行无关重构

---

## 4. 实现原则

1. 本轮重构以“抽离与集中”为主，而不是重新设计。
2. 优先保持行为不变。
3. 如原逻辑依赖全局函数接口，应尽量保持兼容。
4. 新文件中的函数命名应清晰表达职责。
5. 不要为了追求“彻底整理”而额外引入高风险结构变化。

---

## 5. 验收标准

本轮改动后必须满足：

1. 主程序可正常打开。
2. 可正常摆放物品。
3. 可正常保存 scene。
4. 可正常加载 scene。
5. 加载后的场景内容与保存前一致。
6. 默认场景加载功能不受影响。
7. 若存在 local persistence，其行为不受影响。
8. 不得引入新的明显 UI 错误或运行错误。

---

## 6. 本轮交付物

本轮应至少产出：

- `src/scene-storage/scene-storage.js`
- `index.html` 中最小必要的脚本接线改动
- 不超出 scene-storage 范围的结构调整
